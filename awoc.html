A Week Of Culture

Présentation:

A Week Of Culture, alias AWOC, est une application web destiné aux élèves du lycée Gabriel Fauré ayant pour but que chaque élève puisse, chaque semaine, un post sur le sujet de son choix, ayant un rapport à la culture, qu'elle soit numérique, cyber ou autre.

Ce projet sert à créer du lien via un réseau non-officiel du lycée entre les élèves en leur permettant de poster d'eux même et d'interagir avec les posts des autres étudiant via un système de like ou de commentaire.

Création:

Ce projet a été un projet commun des 5 élèves la classe de BTS SIO 2 option SLAM fonctionnant sur le principe de l'agilité. L'un des principes les plus important était l'amélioration continue du site en mettant à jour l'application web régulièrement pour répondre aux besoins des utilisateurs. L'application web a été créé avec le framework PHP Symfony, du JavaScript, de l'HTML, du CSS et l'outil Boostrap. La base de données utilisée étant une base de données SQL relationnelles, nous avons utilisé PHPMyAdmin

Choses Appris
Travailler en mode projet Le projet AWOC a été fait exclusivement en méthode AGILE SCRUM, comprenant des daily scrums, un backlog avec les besoins (users stories) exprimé par le product owner, un travail d'équipe et la remise du produit petit à petit.
Répondre à un besoin Le backlog était incrémenter au fur et à mesure de nouvelle demandes/besoins d'utilisateurs. Lorsque nous choisissions individuellement de travailler sur un de ces besoins, nous devions d'abord le comprendre, savoir de qui est la demande et trouver une solution viable à cette demande.
Apprendre sois-même Nous avons été confrontées à beaucoup de problèmes, notamment liés à l'inconnue. Nous ne savions pas comment répondre a certaine demande ce qui a demander un effort personnel de recherche. Le travail se faisait beaucoup en autonomie, ce qui nous forçait à trouver des solutions nous-même.
Travailler sur des branches / Merge request Le projet a été créé dans un repository GitHub, chaque élève développeur ayant chacun le code, nous ne pouvions pas tous le modifier en même temps. Nous avons donc chacun créer des branches en fonction des users stories que nous prenions permettant d'avoir chacun une instance du code pour finalement les merges, donc les rassembler dans la branche principal en résolvant les conflits possible lié au code d'un des développeurs.
